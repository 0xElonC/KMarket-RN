## 1. 系统架构总览

```
┌──────────────┐     ┌──────────────────┐     ┌──────────────┐
│   用户钱包    │────▶│   KMarketVault   │◀────│ LP 流动性提供 │
│  (EOA/Safe)  │     │  (资金池 + 记账)  │     │   者 (LP)    │
└──────┬───────┘     └───────┬──────────┘     └──────────────┘
       │                     │
       │ SessionKey           │ settleBalances
       │ 授权下注              │ updateLockedBalance
       ▼                     ▼
┌──────────────┐     ┌──────────────────┐
│ SessionKey   │     │ KMarketSettlement│
│ Registry     │     │  (结算引擎)       │
└──────────────┘     └───────┬──────────┘
                             │
                     Oracle EIP-712 签名
                             │
                     ┌───────▼──────────┐
                     │  后端 Sequencer   │
                     │  (链下撮合+签名)   │
                     └──────────────────┘
```

**核心理念**：用户资金统一存入 Vault，下注/结算仅修改链上 mapping 数值（内部记账），不做 ERC20 转账。只有充值和提现才真正移动 USDC。

## 3. KMarketVault — 中心化资金池

### 3.1 状态变量

```
┌─────────────────────────────────────────────────────────────────┐
│ 用户余额层                                                       │
│   balances[user]         用户在 Vault 中的 USDC 余额（含锁仓部分）  │
│   lockedBalance[user]    用户被锁定的金额（已下注但未结算）          │
│   可用余额 = balances - lockedBalance                             │
├─────────────────────────────────────────────────────────────────┤
│ 提现状态层                                                       │
│   fastWithdrawNonce[user]        快速提现 nonce（防重放）          │
│   _slowWithdrawRequests[user]    慢速提现请求                     │
│   _emergencyRequests[user]       紧急提现请求                     │
├─────────────────────────────────────────────────────────────────┤
│ LP 池                                                            │
│   lpPool              LP 池 USDC 总量                            │
│   lpShares[provider]  LP 持有的份额数                             │
│   totalLPShares       LP 份额总供应                               │
├─────────────────────────────────────────────────────────────────┤
│ 全局统计                                                         │
│   totalUserDeposits   所有用户余额之和                             │
│   totalLockedBalance  所有用户锁仓之和                             │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 常量

| 常量 | 值 | 说明 |
|------|-----|------|
| `SLOW_WITHDRAW_DELAY` | 120 秒 | 慢速提现延迟（约 2 个结算周期） |
| `EMERGENCY_WITHDRAW_DELAY` | 7 天 | 紧急提现延迟 |
| `FAST_WITHDRAW_TYPEHASH` | EIP-712 类型哈希 | `FastWithdraw(address user,uint256 amount,uint256 nonce,uint256 deadline)` |

### 3.3 结构体

#### SlowWithdrawRequest
```solidity
struct SlowWithdrawRequest {
    uint256 amount;       // 请求提现金额
    uint256 requestTime;  // 请求发起时间戳
    bool executed;        // 是否已执行
}
```
- 每个用户同一时间只能有 1 个未执行的慢速提现请求
- 判断是否活跃：`amount > 0 && !executed`

#### EmergencyRequest
```solidity
struct EmergencyRequest {
    uint256 requestTime;  // 请求发起时间戳
    bool active;          // 是否活跃
}
```
- 紧急提现忽略 lockedBalance，提走全部余额

### 3.4 事件

| 事件 | 参数 | 触发场景 |
|------|------|----------|
| `Deposited` | user, amount, newBalance | 用户充值 |
| `FastWithdrawn` | user, amount, nonce | 快速提现成功 |
| `SlowWithdrawRequested` | user, amount, readyTime | 发起慢速提现 |
| `SlowWithdrawn` | user, amount | 执行慢速提现 |
| `EmergencyWithdrawRequested` | user, readyTime | 发起紧急提现 |
| `EmergencyWithdrawn` | user, amount | 执行紧急提现 |
| `EmergencyWithdrawCancelled` | user | 取消紧急提现 |
| `BalancesSettled` | userCount, netDelta | 批量结算完成 |
| `LPDeposited` | provider, amount, shares | LP 充值 |
| `LPWithdrawn` | provider, amount, shares | LP 提现 |
| `LiquidityStateUpdated` | lpPool, totalUserDeposits, totalLockedBalance, timestamp | 流动性状态变更 |

### 3.5 错误码

| 错误 | 含义 |
|------|------|
| `ZeroAmount()` | 金额为 0 |
| `InsufficientAvailable(available, requested)` | 可用余额不足 |
| `InsufficientLP(lpPool, required)` | LP 池不够赔付 |
| `SignatureExpired(deadline, currentTime)` | 签名已过期 |
| `InvalidSequencerSig(recovered)` | Sequencer 签名无效 |
| `PendingSlowWithdraw()` | 已有未完成的慢速提现 |
| `NoActiveRequest()` | 没有活跃的提现请求 |
| `TooEarly(readyTime, currentTime)` | 还未到可执行时间 |
| `NoBalance()` | 用户余额为 0 |
| `LengthMismatch()` | 结算数组长度不一致 |
| `InvalidShares()` | LP份额无效 |
| `SlippageExceeded(actual, minimum)` | 滑点保护触发 |

### 3.6 方法详解

#### 3.6.1 deposit(uint256 amount)

```
调用者：任何用户 | 修饰符：nonReentrant, whenNotPaused
```

**流程：**
1. 校验 `amount > 0`
2. `safeTransferFrom`：从用户钱包拉取 USDC 到 Vault
3. `balances[user] += amount`
4. `totalUserDeposits += amount`
5. 发出 `Deposited` 事件

**前置条件：** 用户已对 Vault 合约 approve 了足够的 USDC 额度。

#### 3.6.2 fastWithdraw(uint256 amount, uint256 deadline, bytes sequencerSig)

```
调用者：任何用户 | 修饰符：nonReentrant, whenNotPaused
```

**流程：**
1. 校验 amount > 0、deadline 未过期
2. 计算 `available = balances[user] - lockedBalance[user]`，校验充足
3. 自增 nonce：`fastWithdrawNonce[user]++`
4. 构造 EIP-712 structHash → digest，用 `ECDSA.recover` 恢复签名者
5. 校验签名者拥有 `SEQUENCER_ROLE`
6. 扣减余额，转出 USDC

**设计意图：** 快速提现需要后端 Sequencer 预先确认用户没有未结算注单，然后签发一次性授权。用户持此签名上链立即提现。

**安全保障：**
- EIP-712 结构化签名 → 防 phishing
- nonce 自增 → 防重放
- deadline → 签名有时效

#### 3.6.3 requestSlowWithdraw(uint256 amount)

```
调用者：任何用户 | 修饰符：nonReentrant, whenNotPaused
```

**流程：**
1. 校验 amount > 0、可用余额充足
2. 校验当前没有未执行的慢速提现（`amount > 0 && !executed` 则 revert）
3. 写入新请求 `{amount, requestTime: now, executed: false}`
4. 发出 `SlowWithdrawRequested` 事件

**设计意图：** 无需 Sequencer 签名的自助提现通道。延迟期内后端有时间同步 lockedBalance。

#### 3.6.4 executeSlowWithdraw()

```
调用者：原请求者 | 修饰符：nonReentrant（无 whenNotPaused，暂停时仍可执行已有请求）
```

> 注意：此函数没有 `whenNotPaused`，这意味着即使系统暂停，用户仍然可以执行已经过了延迟期的慢速提现。这是合理的设计——已经通过延迟期的提现不应被暂停阻断。

**流程：**
1. 校验请求存在且未执行
2. 校验当前时间 ≥ requestTime + SLOW_WITHDRAW_DELAY
3. 再次校验可用余额（防止延迟期内被锁仓）
4. `executed = true`，扣减余额，转出 USDC

#### 3.6.5 requestEmergencyWithdraw()

```
调用者：任何有余额的用户 | 修饰符：无（不受 pause 影响）
```

**流程：**
1. 校验用户有余额
2. 写入 `{requestTime: now, active: true}`
3. 发出事件

**设计意图：** 最终自救通道。即使系统完全瘫痪、Sequencer 失联，用户等待 7 天后可无条件取回全部资金。

#### 3.6.6 executeEmergencyWithdraw()

```
调用者：原请求者 | 修饰符：nonReentrant（不受 pause 限制）
```

**流程：**
1. 校验请求活跃、7天已过
2. 缓存 `totalAmount = balances[user]`、`locked = lockedBalance[user]`
3. 清零：`balances = 0`, `lockedBalance = 0`
4. `totalUserDeposits -= totalAmount`
5. `totalLockedBalance -= locked`（带安全检查）
6. `active = false`
7. 转出全部 USDC

**关键特性：** 忽略 lockedBalance，即使有未结算注单也全额提走。7天延迟足够后端结算完所有注单。

#### 3.6.7 cancelEmergencyWithdraw()

**流程：** 将 `active` 设为 `false`。不受 pause 限制。

#### 3.6.8 settleBalances(address[] users, int256[] deltas, uint256[] lockedDeltas)

```
调用者：SETTLEMENT_ROLE | 修饰符：nonReentrant
```

**流程：**
1. 校验三个数组等长
2. 遍历每个用户：
   - 如果 `lockedDeltas[i] > 0`：解锁 → `lockedBalance[user] -= lockedDeltas[i]`, `totalLockedBalance -= lockedDeltas[i]`
   - 如果 `delta > 0`（赢）：`balances[user] += delta`
   - 如果 `delta < 0`（输）：`balances[user] -= |delta|`
   - 累加 netDelta
3. netDelta 与 LP 池对冲：
   - `netDelta > 0`（用户总体赢）：`lpPool -= netDelta`（LP 出钱）
   - `netDelta < 0`（用户总体输）：`lpPool += |netDelta|`（LP 赚钱）

**资金流向核心公式：**
```
用户赢的钱 = LP 输的钱
用户输的钱 = LP 赢的钱
∑ delta_users + delta_LP = 0 （零和）
```

#### 3.6.9 updateLockedBalance(address user, uint256 amount)

```
调用者：SETTLEMENT_ROLE | 无重入锁
```

**流程：** `lockedBalance[user] += amount`, `totalLockedBalance += amount`

**设计意图：** 后端在用户下注后，高频调用此函数锁定用户余额，防止用户在结算前提走下注资金。

#### 3.6.10 lpDeposit(uint256 amount, uint256 minShares)

```
调用者：任何 LP | 修饰符：nonReentrant, whenNotPaused
```

**份额计算：**
```
if totalLPShares == 0 || lpPool == 0:
    shares = amount                          // 初始化模式，1:1
else:
    shares = amount × totalLPShares / lpPool  // 按净值比例
```

**滑点保护：** `shares >= minShares`，否则 revert。防止交易被夹（三明治攻击）。

#### 3.6.11 lpWithdraw(uint256 shares, uint256 minAmount)

```
调用者：持有份额的 LP | 修饰符：nonReentrant, whenNotPaused
```

**赎回计算：**
```
amount = shares × lpPool / totalLPShares
```

**滑点保护：** `amount >= minAmount`，否则 revert。

**注意：** 此函数带 `whenNotPaused`。暂停期间 LP 不能提走流动性，防止市场恐慌时 LP 挤兑导致用户无法提现。

#### 3.6.12 View 函数

| 函数 | 返回值 | 说明 |
|------|--------|------|
| `getAvailableBalance(user)` | `balances[user] - lockedBalance[user]` | 可提现余额 |
| `getAvailableLiquidity()` | `usdc.balanceOf(address(this))` | Vault 合约持有的实际 USDC |
| `getTotalAssets()` | `totalUserDeposits + lpPool` | 账面总资产（不含外部部署） |
| `getSlowWithdrawStatus(user)` | amount, readyTime, canExecute | 慢速提现状态 |
| `getEmergencyStatus(user)` | active, readyTime, timeRemaining | 紧急提现状态 |

| 函数 | 暂停时行为 | 原因 |
|------|-----------|------|
| `deposit` | 🚫 禁止 | 异常期间不接受新资金 |

### 4.1 状态变量

| 变量 | 类型 | 说明 |
|------|------|------|
| `vault` | `IKMarketVault immutable` | 关联的 Vault 合约地址 |

| `sequencer` | `address` | 排序器地址 |
| `admin` | `address` | 管理员地址 |
| `batchNonce` | `uint256` | 全局批次编号，每次 batchSettle 自增 |
| `_poolLedgers[poolId]` | `PoolLedger` | 每个池的账本 |
| `_batchRoots[nonce]` | `bytes32` | 每批次的 Merkle 根 |

### 4.2 常量（TypeHash）

| 常量 | EIP-712 结构 |
|------|-------------|
| `BATCH_SETTLE_TYPEHASH` | `BatchSettle(bytes32 poolId, bytes32 batchRoot, bytes32 stateRoot, uint256 batchNonce, uint256 settlementCount)` |
| `SELF_SETTLE_TYPEHASH` | `SelfSettle(bytes32 poolId, bytes32 userSeal, address user)` |

### 4.3 结构体

#### SettlementItem — 批量结算条目
```solidity
struct SettlementItem {
    address user;       // 用户地址
    int128  netDelta;   // 余额净变动（正=赢, 负=输）
    uint128 lockedDelta;// 本次解锁的锁仓金额
}
```

#### PoolLedger — 池级账本
```solidity
struct PoolLedger {
    uint128 totalBetsReceived;  // 累计收到的下注金额
    uint128 totalPayouts;       // 累计支出（赔付+退还）
    uint128 lockedBets;         // 当前未结算锁定金额
    uint128 lpBalance;          // LP 在该池中的余额（保留字段）
    uint64  lastBatchNonce;     // 最后处理的批次号
    uint64  lastUpdateTime;     // 最后更新时间
}
```

#### PoolDelta — 池级增量
```solidity
struct PoolDelta {
    uint128 totalPayoutsDelta;       // 本批次赔付增量
    uint128 totalBetsReceivedDelta;  // 本批次收注增量
    uint128 lockedBetsDelta;         // 本批次解锁增量
}
```

#### SettleOrder — 自助结算订单
```solidity
struct SettleOrder {
    bytes32 poolId;       // 所属池
    bytes32 tickId;       // 刻度/赔率 ID
    uint128 betAmount;    // 下注金额
    uint128 payoutAmount; // 赔付金额（赢时）
    uint64  expiryTime;   // 订单到期时间
    uint64  timestamp;    // 下注时间
    address user;         // 下注者
    bool    isWin;        // 是否赢了
}
```

### 4.6 方法详解

#### 4.6.1 管理方法

| 方法 | 调用者 | 说明 |
|------|--------|------|
| `setOracle(address _oracle)` | admin | 更新 Oracle 签名地址，发出 `OracleUpdated` 事件 |
| `setSequencer(address _sequencer)` | admin | 更新 Sequencer 地址，发出 `SequencerUpdated` 事件 |
| `transferAdmin(address newAdmin)` | admin | 转移管理员权限 |
| `initializePool(bytes32 poolId)` | admin | 初始化新池，设置 `lastUpdateTime = now` |

#### 4.6.2 batchSettle(...)

```
调用者：sequencer | 修饰符：onlySequencer, nonReentrant
```

**完整签名：**
```solidity
function batchSettle(
    bytes32 poolId,                       // 目标池
    SettlementItem[] calldata settlements,// 结算条目数组
    PoolDelta calldata poolDelta,         // 池级统计增量
    bytes32 batchRoot,                    // 该批次的 Merkle 根
    bytes32 stateRoot,                    // 结算后的状态根
    bytes calldata oracleSignature        // Oracle 的 EIP-712 签名
)
```

**流程：**

```
Step 1: 前置校验
  ├─ settlements 非空
  ├─ pool 已初始化
  └─ 缓存 currentNonce = batchNonce

Step 2: 验证 Oracle EIP-712 签名
  ├─ structHash = keccak256(BATCH_SETTLE_TYPEHASH, poolId, batchRoot, stateRoot, currentNonce, count)
  ├─ digest = hashTypedDataV4(structHash)
  ├─ signer = ECDSA.recover(digest, oracleSignature)
  └─ 校验 signer == oracle

Step 3: 构建 Vault 调用参数
  ├─ 遍历 settlements → users[], deltas[], lockedDeltas[]
  └─ 累计 netDelta

Step 4: 调用 vault.settleBalances(users, deltas, lockedDeltas)
  └─ Vault 内部完成余额调整 + LP 对冲

Step 5: 更新池账本
  ├─ totalBetsReceived += totalBetsReceivedDelta
  ├─ totalPayouts += totalPayoutsDelta
  ├─ lockedBets -= lockedBetsDelta (if > 0)
  ├─ lastBatchNonce = currentNonce
  └─ lastUpdateTime = now

Step 6: 存储 Merkle 根和状态根
  ├─ _batchRoots[currentNonce] = batchRoot
  ├─ _stateRoots[currentNonce] = stateRoot
  └─ batchNonce++

Step 7: 发出事件
  ├─ BatchSettled(poolId, currentNonce, batchRoot, stateRoot, count)
  └─ PoolLedgerUpdated(poolId, ...)
```

**签名验证保障：**
- `batchNonce` 包含在签名中 → 确保每个批次签名唯一
- `poolId` 包含 → 不能跨池重用
- `batchRoot + stateRoot` 包含 → 确保 Oracle 认可具体结算内容

#### 4.6.3 selfSettleAll(...)

```
调用者：任何用户 | 修饰符：nonReentrant
```

**完整签名：**
```solidity
function selfSettleAll(
    bytes32 poolId,                  // 目标池
    SettleOrder[] calldata orders,   // 该用户的所有待结算订单
    bytes calldata oracleSignature,  // Oracle 签名
    bytes32 userSeal                 // 订单集合的密封哈希
)
```

**设计意图：** 当 Sequencer 长时间未处理某用户的结算，用户可以拿着 Oracle 预签名自行提交结算。这是去中心化的安全阀。

**流程：**

```
Step 1: 前置校验
  ├─ orders 非空
  ├─ pool 已初始化
  └─ usedSeals[userSeal] == false（防重放） → 标记为 true

Step 2: 验证 Oracle 签名
  ├─ structHash = keccak256(SELF_SETTLE_TYPEHASH, poolId, userSeal, msg.sender)
  ├─ digest = hashTypedDataV4(structHash)
  └─ 校验 recover(digest, sig) == oracle

Step 3: 验证 userSeal 符合 orders 内容
  ├─ computedSeal = _computeOrdersSeal(orders)
  └─ 校验 computedSeal == userSeal

Step 4: 计算净额
  ├─ 遍历 orders:
  │   ├─ 校验 order.user == msg.sender
  │   ├─ 如果 isWin: netDelta += (payoutAmount - betAmount)
  │   ├─ 如果 !isWin: netDelta -= betAmount
  │   └─ totalLocked += betAmount
  └─ netDelta = 净盈亏, totalLocked = 总解锁量

Step 5: 调用 vault.settleBalances([msg.sender], [netDelta], [totalLocked])

Step 6: 更新池账本
  ├─ totalBetsReceived += totalLocked
  └─ totalPayouts 按净额方向累加

Step 7: 发出事件
```

**防重放机制：** `usedSeals[userSeal]` 确保同一组订单不能被结算两次。

#### 4.6.4 verifyOrder(uint256 batchNonce, bytes32 orderHash, bytes32[] merkleProof)

**用途：** 任何人都可以通过 Merkle Proof 验证某笔订单是否包含在某个批次中。用于链下对账和争议仲裁。

#### 4.6.5 View 函数

| 函数 | 返回值 | 说明 |
|------|--------|------|
| `getPoolLedger(poolId)` | `PoolLedger` | 获取指定池的完整账本 |
| `getBatchRoot(batchNonce)` | `bytes32` | 获取指定批次的 Merkle 根 |

#### 4.6.6 _computeOrdersSeal(SettleOrder[] orders) — 内部函数

**逻辑：**

2. 将所有哈希 `keccak256(abi.encodePacked(hashes))` 组合成一个密封值

## 5. SessionKeyRegistry — 会话密钥注册表

### 5.1 核心概念

用户授权一个临时密钥（SessionKey）代替主钱包签名下注请求。好处：
- 主私钥不暴露给浏览器/前端
- SessionKey 有过期时间、额度限制、权限位掩码
- 用户可随时撤销

### 5.2 状态变量

| 变量 | 说明 |
|------|------|
| `_keys[user][sessionKey]` | SessionKeyData 结构 |
| `_activeKey[user]` | 用户当前活跃的 SessionKey（每个用户只能有 1 个） |
| `isRecorder[addr]` | 被授权记录消费的地址（如 Settlement 合约） |
| `admin` | 管理员 |

### 5.3 事件

| 事件 | 参数 | 触发场景 |
|------|------|----------|
| `SessionKeyAuthorized` | user, sessionKey, expiry, spendingLimit, permissions | 授权新 SessionKey |
| `SessionKeyRevoked` | user, sessionKey | 撤销 SessionKey |
| `SessionKeyUsed` | user, sessionKey, amount, totalSpent | 记录消费 |

### 5.4 权限位掩码

```solidity
PERMISSION_BET    = 1 << 0  // 0b001 = 下注
PERMISSION_CANCEL = 1 << 1  // 0b010 = 取消注单
PERMISSION_SETTLE = 1 << 2  // 0b100 = 结算
```

用法示例：`permissions = PERMISSION_BET | PERMISSION_CANCEL` = 允许下注和取消。

### 5.5 SessionKeyData 结构

```solidity
struct SessionKeyData {
    uint64  expiry;        // 有效期截止时间
    uint64  permissions;   // 权限位掩码
    uint128 spendingLimit; // 最大累计消费额度
    uint128 totalSpent;    // 已消费金额
    bool    active;        // 是否激活
}
```

### 5.6 关键方法

| 方法 | 调用者 | 说明 |
|------|--------|------|
| `authorize(key, expiry, limit, perms)` | 用户 | 授权新 SessionKey，自动撤销旧的 |
| `revoke(key)` | 用户 | 撤销指定 SessionKey |
| `isValid(user, key, requiredPerm)` | 任何人(view) | 检查 key 是否有效+有权限 |
| `recordSpending(user, key, amount)` | Recorder | 记录消费并校验额度 |
| `getKeyData(user, key)` | 任何人(view) | 获取 SessionKeyData 完整结构 |
| `activeKey(user)` | 任何人(view) | 获取用户当前活跃 SessionKey 地址 |
| `setRecorder(recorder, enabled)` | admin | 设置/取消消费记录权限 |
| `transferAdmin(newAdmin)` | admin | 转移管理员权限 |

## 6. 数据流全景图

### 6.1 正常交易周期（充值 → 下注 → 结算 → 提现）

```
时间轴 ──────────────────────────────────────────────────────────▶

T0: 用户充值
    User → approve(vault, 1000) → vault.deposit(1000)
    链上: balances[user] = 1000, totalUserDeposits = 1000
    USDC 流向: User → Vault

T1: 用户授权 SessionKey
    User → registry.authorize(sessionKey, expiry, 500, BET)
    链上: sessionKey 激活，额度 500

T2: 用户下注（链下签名 + 链上锁仓）
    前端:   sessionKey 签名下注请求 → 发送到后端
    后端:   验证签名 → 记入链下订单簿 → 调用 updateLockedBalance(user, 200)
    链上:   lockedBalance[user] = 200, totalLockedBalance = 200
    USDC 流向: 无（仅修改 mapping）

T3: 结算
    后端:   汇总一批用户的盈亏 → Oracle 签名 → Sequencer 调用 batchSettle
    链上:   vault.settleBalances(...)
            用户赢: balances[user] += winAmount
            用户输: balances[user] -= loseAmount
            解锁: lockedBalance[user] -= betAmount
            LP 对冲: lpPool ± netDelta
    USDC 流向: 无（仅修改 mapping）

T4: 用户提现（快速通道）
    前端:   请求 Sequencer 签发提现授权
    后端:   确认无未结算注单 → 用 sequencerPK 签 EIP-712
    用户:   vault.fastWithdraw(800, deadline, sig)
    链上:   balances[user] -= 800, totalUserDeposits -= 800
    USDC 流向: Vault → User
```

### 6.2 异常路径：Sequencer 失联

```
T0: 用户发现 Sequencer 不响应快速提现

T1: 用户发起慢速提现
    vault.requestSlowWithdraw(500)
    → 等待 120 秒

T2: 120 秒后执行
    vault.executeSlowWithdraw()
    → 成功取回 500 USDC

T3: 如果慢速也不行（合约暂停了 deposit/fastWithdraw/requestSlow）
    vault.requestEmergencyWithdraw()
    → 等待 7 天

T4: 7 天后
    vault.executeEmergencyWithdraw()
    → 取回全部余额，忽略 lockedBalance
```

## 7. 业务场景模拟

### 场景 1：Alice 和 Bob 对赌，Alice 赢

```
初始状态:
  balances[Alice] = 1000    lockedBalance[Alice] = 0
  balances[Bob]   = 1000    lockedBalance[Bob]   = 0
  lpPool          = 10000   totalLPShares = 10000

Step 1: 下注锁仓 (updateLockedBalance)
  Alice 下注 500 → lockedBalance[Alice] = 500
  Bob   下注 500 → lockedBalance[Bob]   = 500

Step 2: 市场开奖，Alice 赢
  batchSettle:
    settlements = [
      { user: Alice, netDelta: +500,  lockedDelta: 500 },
      { user: Bob,   netDelta: -500,  lockedDelta: 500 }
    ]

Step 3: settleBalances 执行
  Alice: lockedBalance 500 → 0, balances 1000 → 1500
  Bob:   lockedBalance 500 → 0, balances 1000 → 500
  netDelta = +500 + (-500) = 0
  lpPool 不变 = 10000（零和，LP不介入）

最终状态:
  balances[Alice] = 1500    lockedBalance[Alice] = 0  ✅
  balances[Bob]   = 500     lockedBalance[Bob]   = 0  ✅
  lpPool          = 10000                              ✅
```

### 场景 2：单用户赢，LP 出钱

```
初始状态:
  balances[Alice] = 1000  lpPool = 10000

Step 1: Alice 下注 500, lockedBalance = 500

Step 2: Alice 赢了 300 净利润
  batchSettle:
    settlements = [{ user: Alice, netDelta: +300, lockedDelta: 500 }]

Step 3: settleBalances 执行
  Alice: lockedBalance 500 → 0, balances 1000 → 1300
  netDelta = +300
  lpPool = 10000 - 300 = 9700（LP 承担亏损）

最终:
  balances[Alice] = 1300  ✅
  lpPool          = 9700  ✅ (LP 亏了 300)
```

### 场景 3：多用户输，LP 赚钱

```
初始状态:
  balances[Alice] = 1000, balances[Bob] = 1000, lpPool = 10000

Step 1: 双方各下注 500

Step 2: 双方都输了
  settlements = [
    { user: Alice, netDelta: -500, lockedDelta: 500 },
    { user: Bob,   netDelta: -500, lockedDelta: 500 }
  ]

Step 3: settleBalances
  Alice: balances 1000 → 500
  Bob:   balances 1000 → 500
  netDelta = -1000
  lpPool = 10000 + 1000 = 11000（LP 赚了 1000）

Step 4: LP 提现，享受增值
  LP 持有 10000 shares, totalShares = 10000, lpPool = 11000
  每 share 价值 = 11000 / 10000 = 1.1
  提现 10000 shares → 获得 11000 USDC ✅
```

### 场景 4：自助结算（Sequencer 不在线）

```
初始状态:
  Alice: balances = 1000, lockedBalance = 800（两笔注单各 400）

Step 1: Oracle 预签名确认 Alice 的两笔订单结果
  Order1: betAmount=400, payoutAmount=800, isWin=true
  Order2: betAmount=400, payoutAmount=0,   isWin=false

Step 2: Alice 计算并提交
  userSeal = hash(Order1, Order2)
  Alice → selfSettleAll(poolId, [Order1, Order2], oracleSig, userSeal)

Step 3: 合约执行
  netDelta = (800-400) + (-400) = 0
  totalLocked = 400 + 400 = 800
  → vault.settleBalances([Alice], [0], [800])
  → lockedBalance 800 → 0, balances 不变

Step 4: Alice 提现不再被锁仓阻挡
  可用余额 = 1000 - 0 = 1000
  Alice 可以自由提现 ✅
```

### 场景 5：LP 份额价值随盈亏波动

```
T0: LP-A 存入 10000 → shares = 10000, lpPool = 10000
    净值: 1 share = 1 USDC

T1: 用户总输 2000，lpPool = 12000
    净值: 1 share = 12000/10000 = 1.2 USDC

T2: LP-B 存入 6000
    shares = 6000 × 10000 / 12000 = 5000
    lpPool = 18000, totalShares = 15000
    净值: 1 share = 18000/15000 = 1.2 USDC ✅ (LP-B 不享受之前的利润)

T3: 用户总赢 3000，lpPool = 15000
    净值: 1 share = 15000/15000 = 1.0 USDC

T4: LP-A 提走全部
    amount = 10000 × 15000 / 15000 = 10000
    LP-A 收回原始投入（利润被用户赢走了）
    lpPool = 5000, totalShares = 5000

T5: LP-B 提走全部
    amount = 5000 × 5000 / 5000 = 5000
    LP-B 亏了 1000 USDC（存入 6000，取回 5000）
```

## 8. 安全机制一览

| 机制 | 实现方式 | 防御目标 |
|------|---------|---------|
| 重入保护 | `ReentrancyGuard` | 重入攻击 |
| 安全转账 | `SafeERC20` | 非标准 ERC20 返回值 |
| 角色权限 | `AccessControl` (Vault) / 手写 modifier (Settlement) | 未授权访问 |
| 暂停机制 | `Pausable` + `whenNotPaused` | 紧急停机 |
| EIP-712 签名 | `EIP712` + `ECDSA.recover` | 签名伪造 / phishing |
| 快速提现防重放 | `fastWithdrawNonce[user]++` | 签名重放 |
| 自助结算防重放 | `usedSeals[seal] = true` | 结算重放 |
| 批量结算防重放 | `batchNonce` 嵌入签名 | 批次重放 |
| 签名有效期 | `deadline` 参数 | 过期签名被使用 |
| 紧急提现 | 7天延迟 + 不受 pause 限制 | 系统失控时用户自救 |
| LP 滑点保护 | `minShares` / `minAmount` 参数 | 三明治攻击 / MEV |
| Checked Arithmetic | Solidity 0.8 默认 | 溢出/下溢 |

## 9. 全局不变量

以下是合约运行过程中应始终成立的数学关系：

```
1. ∀ user: lockedBalance[user] <= balances[user]
   违反时 getAvailableBalance() 会下溢 revert

2. totalLockedBalance = ∑ lockedBalance[user]
   通过 updateLockedBalance / settleBalances / executeEmergencyWithdraw 维护

3. totalUserDeposits = ∑ balances[user]
   通过 deposit / fastWithdraw / slowWithdraw / settleBalances / emergencyWithdraw 维护

4. usdc.balanceOf(vault) >= totalUserDeposits + lpPool

5. ∑ lpShares[provider] = totalLPShares

6. settleBalances 中: ∑ deltas 由 lpPool 对冲 → 零和
   即: ∑ delta_users + delta_LP = 0

7. batchNonce 严格单调递增，每次 batchSettle +1

8. 每个 userSeal 最多被 selfSettleAll 使用一次
```
